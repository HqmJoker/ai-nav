✨ 1: OpenAI Codex CLI

OpenAI Codex CLI是轻量级终端编码助手，具备ChatGPT推理能力，可运行代码、操作文件并迭代，实现代码版本控制。

OpenAI Codex CLI 是一个轻量级的编码助手，旨在让你在终端中就能利用 ChatGPT 级别的推理能力，并且可以直接运行代码、操作文件以及迭代，所有的操作都受到版本控制。

OpenAI Codex CLI 具有以下特点：

  • 终端环境集成： 为那些习惯在终端中工作的开发者设计。
  • AI 驱动： 利用 OpenAI 的 Codex 模型进行代码生成、解释和修改等操作。
  • 代码执行能力： 能够实际运行生成的代码，安装依赖项，并展示运行结果。
  • 版本控制： 所有更改都可以在版本控制下进行，方便管理。
  • 安全沙箱： 代码执行在沙箱环境中，防止恶意代码影响系统。
  • 开源： 完全开源，允许用户查看和贡献代码。
  • 零配置： 只需要 OpenAI API 密钥即可使用。
  • 多种批准模式： 提供不同的批准模式，允许用户控制代理的自主程度。

OpenAI Codex CLI 适用于需要在终端中快速进行代码生成、修改、理解和安全检查等任务的开发者，它将 AI 的能力带到终端，提高开发效率。

地址：https://github.com/openai/codex

✨ 2: GPT-4.1 Prompting Guide

GPT-4.1 Prompting Guide是OpenAI提供的指南，旨在帮助开发者充分利用GPT-4.1模型的改进能力，包括 agentic 工作流、长上下文处理和指令遵循等方面。

GPT-4.1 Prompting 指南 概述

这份指南汇集了 OpenAI 内部大量测试后总结的 prompt 技巧，旨在帮助开发者充分利用 GPT-4.1 模型家族的强大功能，尤其是在编码、指令遵循和处理长上下文方面。

  1. 指令的迁移和精确性：

    • GPT-4.1 更严格地遵循指令，因此需要更明确和精确的 prompt。以往的模型可能会更自由地推断意图，但 GPT-4.1 需要更直接的指示。
    • 如果模型行为不符合预期，只需用一句话明确说明所需行为，通常就能纠正。

  2. Agentic 工作流：

    • GPT-4.1 擅长构建 agentic 工作流。建议在所有 agent prompt 中包含以下三种提醒：
      • 持久性 (Persistence): 确保模型理解进入多轮对话，防止过早放弃控制权。 例：“请持续进行，直到用户查询完全解决。”
      • 工具调用 (Tool-calling): 鼓励模型充分利用工具，减少幻觉。 例：“如果不确定文件内容或代码库结构，请使用工具读取文件，不要猜测。”
      • 规划 (Planning, 可选): 确保模型在每次工具调用前进行规划和反思。 例：“在每次函数调用前必须进行广泛的规划，并对前一次函数调用的结果进行广泛的反思。”
    • 使用 OpenAI API 的 tools 字段传递工具，而不是手动将工具描述注入 prompt。

  3. 长上下文处理：

    • GPT-4.1 具有强大的 1M token 输入上下文窗口，适用于结构化文档解析、重排序、选择相关信息和多跳推理等任务。
    • 指令的最佳位置是在提供的上下文的开头和结尾。如果只想包含一次指令，放在上下文的上方效果更好。
    • 考虑回答问题所需的外部与内部知识的结合。有时需要模型使用自身知识连接概念或逻辑跳转，而有时最好只使用提供的上下文。

  4. Chain of Thought (CoT)：

    • 虽然 GPT-4.1 不是一个推理模型，但引导模型逐步思考（CoT）可以有效地将问题分解成更易于管理的部分。
    • 从简单的 CoT 指令开始，例如：“首先，仔细思考回答查询所需的文档。然后，打印出每个文档的标题和 ID。然后，将 ID 格式化为列表。”
    • 可以通过审计失败的例子和评估，并针对系统性的规划和推理错误给出更明确的指令来改进 CoT prompt 。

  5. 指令遵循：

    • GPT-4.1 在指令遵循方面表现出色，可以精确地控制输出。
    • 推荐的工作流程：
      • 从一个总体的“响应规则”或“指令”部分开始，提供高级指导。
      • 如果想改变更具体的行为，添加一个部分来指定该类别的更多细节，例如“# 示例短语”。
      • 如果有希望模型在其工作流程中遵循的特定步骤，添加一个有序列表并指示模型遵循这些步骤。

  6. 通用建议：

    • Prompt 结构：定义角色和目标，提供明确的指令，指定推理步骤、输出格式，并提供示例。
    • 分隔符：推荐使用 Markdown 或 XML 格式，尤其是在处理大量文档时。JSON 也可以，但在代码上下文中更合适。
    • 局限性：模型可能对生成非常长、重复的输出有所抵触。如果需要，应强烈指示模型完整输出信息。同时，并行工具调用可能会出错，建议测试并考虑禁用并行调用。

  7. Diffs 生成

    • GPT-4.1 在 diff 功能方面有了显著改进。文档中提供了一种推荐的 diff 格式，该模型经过了广泛的训练。
    • apply_patch 工具是进行代码文件添加、删除、移动或编辑。

地址：https://cookbook.openai.com/examples/gpt4-1_prompting_guide

✨ 3: DroidRun

DroidRun是一个强大的框架，通过LLM代理用自然语言控制安卓设备，实现自动化操作。

DroidRun 是一个强大的框架，它允许你通过大语言模型 (LLM) 智能体来控制 Android 设备。简单来说，它能让你用自然语言命令自动执行 Android 设备上的操作。

  • 自然语言控制: 使用自然语言命令控制 Android 设备。
  • 多LLM支持: 支持多种 LLM 服务提供商，包括 OpenAI, Anthropic 和 Gemini。
  • 易于使用的CLI: 提供简单的命令行界面。
  • 可扩展的Python API: 提供 Python API，方便用户自定义自动化脚本。
  • 截图分析: 具备截图分析能力，以便更好地理解设备上的视觉信息。

地址：https://github.com/droidrun/droidrun

✨ 4: MCP Gateway

MCP Gateway是LLM的中间层，集中管理MCP服务器，拦截请求和响应，并提供统一的接口，增强AI安全性。

MCP Gateway 是一个模型上下文协议 (MCP) 服务器的中间层解决方案，它集中化并增强了你的 AI 基础设施。本质上，它扮演着一个代理的角色，位于大型语言模型 (LLM) 和其他 MCP 服务器之间，提供以下关键功能：

  • 配置管理: 读取并管理 mcp.json 文件中的 MCP 服务器配置。
  • 生命周期管理: 管理配置的 MCP 服务器的生命周期。
  • 安全保障 (Guardrails): 拦截请求和响应，对敏感信息进行脱敏处理，防止数据泄露。
  • 统一接口: 为所有代理的 MCP 服务器提供统一的接口，方便发现和交互。
  • 插件机制: 支持多种插件（如 basic, presidio, lasso）来增强安全性和功能，例如通过 Presidio 进行 PII 识别和脱敏。

地址：https://github.com/lasso-security/mcp-gateway
